## 1. 테스트 범위 및 전제조건

### 범위

Chat-Ki-S MVP의 **전체 사용자 흐름**:

```
포털 탭 진입 → 세션 인식 → 질문 입력 → 답변 반환
→ 출처 표시 → 최소화 → 플로팅 전환 → 피드백 제출
```

### 전제조건

| 항목 | 값 |
|------|----|
| Frontend URL | `http://localhost:3000` |
| Backend URL | `http://localhost:4000` |
| AI Service URL | `http://localhost:8001` |
| 테스트 계정 | `employeeId: TEST001`, `password: testpass123` |
| AUTH_MODE | `local` (개발) |
| DB | PostgreSQL (Docker) |

### 테스트 실행 환경

```bash
# 전체 스택 실행
docker-compose up -d   # PostgreSQL + ChromaDB
npm run dev            # Frontend(3000) + Backend(4000) 동시 실행
# AI 서비스는 apps/ai에서 별도 실행

# E2E 테스트 (Playwright)
npx playwright test --project=chromium

# 모바일 E2E
npx playwright test --project="Mobile Safari"

# 단위/통합 테스트 (Jest)
npm run test
```

## 2. 흐름 시나리오 — 정상 케이스

### TS-01: 포털 탭 진입 및 세션 인식

**목적**: 개발 모드에서 로그인 후 채팅 화면으로 올바르게 진입되는지 확인

| 단계 | 행동 | 기대 결과 |
|------|------|-----------|
| 1 | `http://localhost:3000/login` 접근 | 로그인 폼 표시 (`[name=employeeId]`, `[name=password]`) |
| 2 | `TEST001` / `testpass123` 입력 후 제출 | `POST /api/auth/login` 호출, 200 OK, JWT 발급 |
| 3 | 자동 리다이렉트 | `/chat` 화면으로 이동 |
| 4 | `localStorage.getItem('accessToken')` 확인 | JWT 토큰 저장됨 |
| 5 | 헤더 확인 | "Chat-Ki-S / 키움증권 온보딩 챗봇" 표시 |
| 6 | 채팅 영역 확인 | 빈 화면에 환영 메시지 + 안내 문구 표시 |

**관련 E2E**: `auth.spec.ts` — E2E-AUTH-01, E2E-AUTH-04

---

### TS-02: 질문 입력 및 AI 답변 반환

**목적**: 사용자가 질문 입력 시 백엔드 → AI 서비스 → 답변 반환 흐름 동작 확인

| 단계 | 행동 | 기대 결과 |
|------|------|-----------|
| 1 | `/chat` 접근 (로그인 상태) | 채팅 입력창 활성화 상태 |
| 2 | 입력창에 "연차 신청 방법을 알려주세요" 입력 | 입력창에 텍스트 표시 |
| 3 | Enter 키 또는 전송 버튼 클릭 | 사용자 메시지 채팅 목록에 즉시 추가 (파란 버블) |
| 4 | 전송 중 상태 | 입력창 비활성화, 전송 버튼 비활성화 |
| 5 | AI 응답 대기 | 로딩 인디케이터 표시 |
| 6 | 응답 수신 | AI 메시지 채팅 목록에 추가 (회색 버블, "AI" 아바타) |
| 7 | 응답 후 | 입력창 재활성화, 다음 입력 가능 |

**API 확인 포인트**:
- `POST /api/chat` → `{ content, mode: "rag", conversationId? }` 전송
- 응답: `{ conversationId, message: { id, role: "assistant", content } }`
- `conversationId` localStorage 또는 state에 유지되는지 확인

**관련 E2E**: `chat.spec.ts` — E2E-CHAT-01, E2E-CHAT-02

### TS-03: 대화 상태 유지 (conversationId 연속성)

**목적**: 여러 질문을 보내도 같은 대화 세션이 유지되는지 확인

| 단계 | 행동 | 기대 결과 |
|------|------|-----------|
| 1 | 첫 질문 전송 | conversationId 생성됨 |
| 2 | 두 번째 질문 전송 | 동일 conversationId로 `POST /api/chat` 호출 |
| 3 | 세 번째 질문 전송 | 동일 conversationId 유지 |
| 4 | Network 탭 확인 | 2번째 요청부터 `conversationId` 필드 포함 |

### TS-04: 최소화 → 플로팅 챗 전환

**목적**: 채팅창 최소화 시 우하단 플로팅 버튼으로 전환되는지 확인

> **현재 상태**: 플로팅 컴포넌트 미구현 (dev-priorities #5). 구현 완료 후 테스트 실행.

| 단계 | 행동 | 기대 결과 |
|------|------|-----------|
| 1 | 채팅 화면에서 최소화 버튼 클릭 | 풀페이지 챗 숨김 |
| 2 | 화면 전환 | 우하단에 플로팅 버튼(아이콘) 표시 |
| 3 | 플로팅 버튼 클릭 | 채팅창 다시 풀페이지로 복원 |
| 4 | 복원 후 대화 목록 확인 | 최소화 전 메시지 그대로 유지 |

**검증 포인트**: 최소화/복원 후 conversationId 및 message 배열 유지 여부

## 4. 버그 체크리스트
